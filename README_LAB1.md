## Дисциплина

Архитектура информационных систем / Программная инженерия, 1 курс, 2 семестр магистратуры ФИИТ МАИ.

## Вариант

**Вариант 10 — Планирование задач, аналог LeaderTask**  
Сайт-ориентир: https://www.leadertask.ru/

## Цель работы

Получить навык описания архитектуры программной системы в стиле Architecture as Code и проектирования системы сверху вниз с использованием Structurizr DSL и C4-модели.

## Текст индивидуального задания

Приложение должно содержать следующие данные:

- цель;
- задача;
- исполнитель.

Необходимо реализовать API:

- создание нового пользователя;
- поиск пользователя по логину;
- поиск пользователя по маске имени и фамилии;
- создание новой цели;
- получение списка всех целей;
- создание новой задачи на пути к цели;
- получение всех задач цели;
- изменение статуса задачи в цели.

## Описание предметной области

Разрабатываемая система представляет собой веб-приложение для планирования целей и задач. Пользователь может создавать цели, разбивать их на задачи, назначать исполнителей и отслеживать изменение статусов задач. Система ориентирована на личное и командное планирование.

В рамках лабораторной работы рассматривается первая версия архитектуры системы. Основной backend реализуется как модульный REST API сервис, а не как набор отдельных микросервисов. Такой подход выбран для упрощения реализации, уменьшения сетевого взаимодействия между компонентами и сохранения возможности дальнейшего развития системы.

## Основные сущности

### Пользователь

Пользователь системы, который может быть исполнителем задачи, автором цели, менеджером или администратором.

Основные поля:

- идентификатор пользователя;
- логин;
- имя;
- фамилия;
- email;
- телефон;
- роль.

### Цель

Цель, к которой привязаны задачи.

Основные поля:

- идентификатор цели;
- название;
- описание;
- автор цели;
- дата создания;
- статус цели.

### Задача

Задача, созданная на пути к достижению цели.

Основные поля:

- идентификатор задачи;
- идентификатор цели;
- название;
- описание;
- исполнитель;
- автор задачи;
- статус;
- срок выполнения.

## Роли пользователей

### Исполнитель

Базовый пользователь системы. Может создать собственную учетную запись, искать других пользователей, просматривать доступные ему цели и задачи, а также изменять статус задач, где он назначен исполнителем.

### Менеджер

Пользователь, который управляет целями и задачами группы. Может создавать цели, создавать задачи в целях, назначать исполнителей, просматривать задачи по своим целям и менять статусы задач.

### Администратор

Техническая роль для сопровождения системы. Может управлять учетными записями пользователей и просматривать данные системы для поддержки и диагностики.

## Внешние системы

### Email-сервер

Используется для отправки уведомлений пользователям о создании задач, назначении исполнителя и изменении статуса задачи.

### SMS-сервер

Используется для отправки срочных уведомлений исполнителям, например при назначении важной задачи или изменении срока выполнения.

## Основные варианты использования

### Создание нового пользователя

Пользователь отправляет регистрационные данные через веб-интерфейс. API проверяет корректность данных, проверяет уникальность логина и сохраняет нового пользователя в базе данных.

### Поиск пользователя по логину

Пользователь вводит логин. API выполняет точный поиск по логину и возвращает найденную учетную запись.

### Поиск пользователя по маске имени и фамилии

Пользователь вводит часть имени или фамилии. API выполняет поиск по маске и возвращает список подходящих пользователей.

### Создание новой цели

Менеджер или исполнитель создает цель. API сохраняет цель и связывает ее с автором.

### Получение списка всех целей

Пользователь запрашивает список целей. API возвращает цели, доступные пользователю согласно его роли и правам доступа.

### Создание новой задачи на пути к цели

Менеджер или автор цели создает задачу, указывает цель, описание, срок и исполнителя. API сохраняет задачу, связывает ее с целью и передает событие в фоновый обработчик уведомлений.

### Получение всех задач цели

Пользователь открывает цель. API возвращает список задач, относящихся к выбранной цели.

### Изменение статуса задачи в цели

Исполнитель или менеджер изменяет статус задачи. API проверяет права пользователя, обновляет статус в базе данных и передает событие в фоновый обработчик уведомлений.

## Контейнеры системы

### Web Application

Клиентское веб-приложение. Предоставляет интерфейс для регистрации, поиска пользователей, создания целей, создания задач и изменения статусов.

Технология: React / TypeScript.

### Task Planning API

Основное серверное приложение, предоставляющее REST API для клиентского приложения. Выполняет обработку HTTP-запросов, проверку входных данных, аутентификацию, работу с пользователями, целями и задачами.

Технология: C++ / Yandex Userver, REST API.

Внутри Task Planning API выделяются логические модули:

- User Module — создание пользователя, поиск по логину, поиск по имени и фамилии;
- Goal Module — создание цели и получение списка целей;
- Task Module — создание задачи в цели, получение задач цели, изменение статуса задачи;
- Auth Module — проверка учетных данных, ролей и прав доступа;
- Notification Producer — формирование событий для отправки уведомлений.

Эти модули не являются отдельными развертываемыми сервисами в первой версии системы. Они выделены логически внутри одного backend-приложения, чтобы упростить реализацию и при этом сохранить разделение ответственности.

### Notification Worker

Фоновый обработчик событий задач. Получает события о создании или изменении задач и отправляет уведомления через внешние системы.

Технология: C++ / Yandex Userver Worker.

### Database

Единая реляционная база данных для хранения пользователей, целей, задач, статусов и прав доступа.

Технология: PostgreSQL.

## Основные взаимодействия контейнеров

1. Пользователь работает с системой через Web Application.
2. Web Application отправляет HTTPS/REST-запросы в Task Planning API.
3. Task Planning API обрабатывает запросы через логические модули User Module, Goal Module и Task Module.
4. Task Planning API читает и записывает данные в PostgreSQL.
5. При создании задачи или изменении ее статуса Task Planning API передает событие в Notification Worker.
6. Notification Worker отправляет уведомления через Email Server или SMS Server.

## Архитектурно значимый сценарий для Dynamic diagram

Для динамической диаграммы выбран сценарий создания новой задачи на пути к цели, потому что в нем участвуют пользовательский интерфейс, основной REST API, база данных, фоновый обработчик уведомлений и внешние системы отправки уведомлений.

Последовательность:

1. Менеджер заполняет форму создания задачи.
2. Web Application отправляет запрос в Task Planning API.
3. Task Planning API проверяет существование цели, исполнителя и права доступа.
4. Task Planning API сохраняет задачу в PostgreSQL.
5. Task Planning API передает событие TaskCreated в Notification Worker.
6. Notification Worker отправляет email-уведомление исполнителю через Email Server.
7. Notification Worker при необходимости отправляет SMS-уведомление через SMS Server.

## Обоснование архитектурного решения

Для первой версии системы выбран модульный монолит на C++ / Yandex Userver. Это позволяет реализовать все API из варианта в одном backend-приложении, упростить транзакции и подключение к базе данных, но сохранить логическое разделение по предметным областям.

Выделение отдельных микросервисов User Service, Goal Service и Task Service на данном этапе не требуется, потому что система имеет ограниченный набор функций, а основные операции тесно связаны между собой. При росте нагрузки или усложнении требований логические модули можно будет выделить в самостоятельные сервисы.
